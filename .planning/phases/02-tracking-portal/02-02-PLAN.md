<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| share link (URL token) -> server | An untrusted, attacker-controllable token string crosses into the server at /track/[token] |
| share-view -> Nexus data | A valid token grants postcode-gate-free read of one consignment's status |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Spoofing | verifyShareToken — forged token | mitigate | HMAC-SHA256 over the payload with a server-only secret; signature recomputed and compared with crypto.timingSafeEqual; any mismatch -> null. Tested with an altered-signature case |
| T-02-05 | Tampering | verifyShareToken — altered consignment number in payload | mitigate | Signature covers the full payload; changing `c` invalidates the sig -> null. Tested with an altered-payload case |
| T-02-06 | Information Disclosure | share token contents | mitigate | Token encodes only `{c, exp}` — never the postcode or address; test asserts payload keys are exactly ['c','exp'] and codec source contains no 'postcode' reference |
| T-02-07 | Elevation of Privilege | lookupForShare bypasses the postcode gate | accept | The HMAC token IS the authorisation — it is only minted server-side after a successful postcode-gated lookup (Plan 04). Bypassing the gate on the share path is the intended design (D-12); the short expiry (24h) limits the replay window |
| T-02-08 | Denial of Service / Replay | replaying a captured valid token | accept | Embedded `exp` bounds the replay window to the TTL; share pages are read-only status views with no postcode/PII exposed, so replay value is low. Tested via the expired-token case |
</threat_model>
