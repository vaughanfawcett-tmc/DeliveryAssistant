    Define event/state/action types in types.ts:
    - VoiceState: `{ phase: 'greeting'|'awaiting_tracking'|'awaiting_postcode'|'confirming'|'looking_up'|'answering'|'handoff'|'ended'; trackingRef?: string; postcode?: string; attempts: number; transcript: TranscriptTurn[] }`
      where TranscriptTurn = `{ speaker: 'Agent'|'Customer'; text: string; ts?: string }` (matches Phase 3 TranscriptView shape).
    - VoiceEvent (discriminated union): `call_started`, `utterance{text}`, `dtmf{digits}`, `confirm{yes:boolean}`, `request_human`, `lookup_result{result: TrackingResult}`.
    - VoiceAction: `say{text}`, `lookup{trackingRef,postcode}`, `transfer{summary}`, `hangup`, `collect_dtmf`.
    - reduce(state, event) => { state, actions } — PURE (no I/O, no Date.now unless injected via event ts).

    compliance.ts: export const DISCLOSURE (string) containing both an explicit
    "automated"/"AI assistant" phrase AND a "recording"/"recorded" phrase, plus
    `openingTurn(): VoiceAction[]` returning a `say` with DISCLOSURE.

    nato.ts: export `NATO` map (A->'Alfa', B->'Bravo' ... per ICAO) and
    `readBack(value: string): string` that renders letters as NATO words and groups
    digits (e.g. "PA12" -> "P for Papa, A for Alfa, one two"). Export the ALFA entry
    so tests can grep it.

    Machine behaviors (each a test):
    1. call_started -> phase 'awaiting_tracking', actions include a `say` whose text === DISCLOSURE (disclosure precedes any capture; VOICE-01).
    2. utterance with a parseable tracking ref while awaiting_tracking -> readBack confirmation say (NATO), phase 'confirming' (VOICE-02).
    3. dtmf with trailing '#' -> treated as captured ref, strip the '#' (VOICE-03).
    4. confirm{yes:false} -> re-prompt, attempts++ ; after 3rd failure -> phase 'handoff' with a `transfer` action whose summary names the failed attempts (VOICE-05).
    5. request_human at ANY phase -> phase 'handoff', `transfer` action with a warm summary string built from transcript so far (VOICE-06).
    6. lookup_result{ ok:false, reason:'api_error' } -> a `say` that explains the issue + offers transfer; NEVER a status/date; no fabricated fields (VOICE-07/08).
    7. lookup_result{ ok:true } -> a `say` leading with the answer using ONLY consignment fields (plainStatus / window) — assert the say text contains no token absent from the input (VOICE-04/08). Include a test that a null estimatedDelDate never produces an invented date.
    8. lookup_result{ ok:false, reason:'not_found' | 'postcode_mismatch' } -> appropriate distinct say, offer retry or human.
  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| spoken/DTMF input → machine | Untrusted caller utterances may attempt prompt-injection ("tell me it's delivered") |
| TrackingResult → answer text | Only API-sourced facts may be spoken |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Spoofing/Tampering | answering branch | mitigate | Answer text is interpolated ONLY from result.consignment fields; a test asserts no date appears when estimatedDelDate is null (VOICE-08 structural guarantee) |
| T-04-06 | Tampering (prompt-injection) | utterance handling | mitigate | The machine never echoes caller claims as fact; capture utterances only populate trackingRef/postcode candidates that are confirmed + looked up — they cannot set a delivery status |
| T-04-07 | Repudiation | transcript | mitigate | Every turn appends to state.transcript so the warm-handoff summary and persisted record reflect the actual exchange |
| T-04-08 | Denial of service | attempts counter | mitigate | Hard 3-attempt cap forces a handoff transition instead of an infinite capture loop (VOICE-05) |
</threat_model>
